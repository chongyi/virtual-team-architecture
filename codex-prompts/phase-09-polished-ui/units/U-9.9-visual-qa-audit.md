# U-9.9 全应用视觉一致性审计与修复

## 目标 (Goal)

遍历所有页面和组件，审计视觉一致性（颜色/字号/间距/圆角/阴影是否全部使用设计 Token）、交互一致性（同一操作在不同页面是否有相同的交互模式）、信息架构一致性（页面层级/导航路径是否合理），修复所有发现的不一致问题。

## 上下文 (Context)

- 前置：U-9.8（深色模式和无障碍完成）
- Phase 9 前 8 个单元逐个重塑了各个功能区域，可能存在跨单元的风格漂移

## 审计清单

### 一、设计 Token 合规审计

逐页检查以下违规：

- [ ] **硬编码颜色**：搜索代码中 `Color(0x` 或 `Colors.`，确认仅 `design_tokens.dart` 中有颜色定义
- [ ] **硬编码字号**：搜索 `fontSize:`，确认使用 Typography Token 或 `AppTypography.xxx.style`
- [ ] **硬编码间距**：搜索 `EdgeInsets` 和 `SizedBox`，确认间距值是 8 的倍数且使用 `AppTokens.spacingXxx`
- [ ] **硬编码圆角**：搜索 `BorderRadius.circular(`，确认使用 `AppTokens.radiusXxx`
- [ ] **主题未引用**：确认所有页面通过 `Theme.of(context)` 获取颜色/字号，而非直接引用 Token（允许直接引用 Token 作为备选）

**工具辅助**：运行 `grep -rn "Color(0x" apps/flutter/lib/ | grep -v design_tokens.dart` 查找硬编码颜色。

### 二、组件使用审计

逐页检查以下违规：

- [ ] **未使用 AppButton**：搜索 `ElevatedButton`、`TextButton`、`OutlinedButton`，确认已替换为 `AppButton.primary/secondary/text`
- [ ] **未使用 AppAvatar**：搜索 `CircleAvatar`、包含 "avatar" 的手写组件，确认已替换为 `AppAvatar`
- [ ] **未使用 AppEmptyState**：搜索 `"暂无"` 或 `"没有"` 等空状态文案，确认使用了统一组件
- [ ] **未使用 AppLoading**：搜索 `CircularProgressIndicator` 且未包裹在 `AppLoading` 中
- [ ] **未使用 AppErrorWidget**：搜索 `"错误"` 或 `"失败"` 的错误提示，确认使用了统一组件

### 三、页面级别审计

逐页打开以下页面，检查视觉质量：

**登录/注册页**：
- [ ] 表单居中对齐，品牌 Logo 或应用名在顶部
- [ ] 输入框有图标前缀（用户/密码）
- [ ] 登录按钮全宽 + 主色填充

**消息页（首页/核心页面）**：
- [ ] 未读消息角标清晰
- [ ] 输入框固定底部
- [ ] 消息气泡三种样式正确（自己/他人/VE）
- [ ] 空状态（无消息时）正确显示

**通讯录/频道**：
- [ ] 列表滚动流畅
- [ ] 在线状态圆点大小一致
- [ ] 搜索栏在所有需要搜索的页面出现

**协作工具**（文档/表格/看板/审批/日程）：
- [ ] 卡片/列表项风格一致（尺寸/间距/圆角/阴影）
- [ ] FAB 按钮风格一致（颜色/位置/图标大小）
- [ ] 空状态语义正确（对应工具的空状态图标和文案）

**设置/个人页**：
- [ ] 分区标题风格统一（Section Header）
- [ ] 开关/选择器/滑块风格一致
- [ ] 退出登录按钮红色（与主题 Danger 色保持一致）

### 四、深色模式专项审计

在深色模式下检查：

- [ ] 所有文字可读（无文字与背景色相近的情况）
- [ ] 消息气泡两侧对比度足够（自己的气泡在深色背景下是否刺眼）
- [ ] 输入框/卡片/弹窗的背景色合理区分
- [ ] 在线状态圆点在深色背景仍可辨识
- [ ] 代码块在深色模式下可读性
- [ ] 图片在深色模式下没有白边（透明 PNG 可能的问题）
- [ ] 骨架屏/加载指示器在深色模式下可见

### 五、响应式审计

在每个断点检查：

- [ ] **移动端（375px 宽度）**：所有文字不溢出，按钮触控区域 > 48px
- [ ] **平板（768px 宽度）**：侧边栏或底部导航正常，内容区利用充分
- [ ] **桌面（1280px 宽度）**：侧边栏常驻，内容区最大宽度合理（不超过 900px 以避免行过长）

### 六、性能审计

- [ ] 消息列表使用 `ListView.builder`（而非 `ListView(children: [...])`）
- [ ] 大列表使用 `itemExtent` 或 `prototypeItem` 优化滚动
- [ ] 图片列表使用 `CachedNetworkImage` 缓存
- [ ] 页面切换不重建已存在的页面（IndexedStack / AutomaticKeepAliveClientMixin）
- [ ] 在 `--profile` 模式下滚动无卡顿（肉眼检查）

## 修复原则

- 发现硬编码 Token → 替换为设计 Token 引用
- 发现手写组件 → 替换为全局组件（AppButton/AppAvatar/...）
- 发现风格不一致 → 以 U-9.1 定义的设计系统为准
- 发现可读性问题 → 调整颜色 Token（不修改单个页面的一处颜色）
- 发现性能问题 → 替换为高效组件（ListView.builder 等）

## 约束 (Constraints)

- 修复过程中不破坏任何现有功能
- 审计报告以注释形式记录在 audit 文件中（供后续维护参考）
- 每个审计项目修复后单独 commit，便于追溯
