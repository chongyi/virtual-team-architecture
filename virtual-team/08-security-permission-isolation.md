# 08. 安全、权限与隔离

## 1. 安全目标

VT 的安全模型需要同时覆盖：

- SaaS 多租户隔离。
- 组织和成员权限。
- 虚拟员工权限。
- 消息与工作上下文访问控制。
- 工具调用审批。
- WorkEnvironmentNode 沙箱。
- 云工作环境隔离。
- 审计和追责。

Agent 可以执行真实操作，因此安全边界必须产品化，而不是只依赖用户懂技术。

## 2. 多层权限模型

权限至少分为以下层级：

| 层级 | 说明 |
|------|------|
| 租户权限 | 数据和资源隔离的最高边界 |
| 组织权限 | 组织内成员、频道、任务和虚拟员工管理 |
| 会话权限 | 私聊、群组、频道中的消息访问 |
| 虚拟员工权限 | 某个 VTE 可以服务哪些组织和任务 |
| 工作上下文权限 | 谁可以查看、更新、接管或关闭某项工作 |
| 工具权限 | 哪些工具可见、可调用、需审批 |
| 工作环境权限 | 哪些节点和空间可被哪个 VTE 使用 |

## 3. 虚拟员工权限

虚拟员工需要像真实成员一样受权限约束。

系统需要控制：

- 能否加入某组织。
- 能否进入某频道。
- 能否读取历史消息。
- 能否查看附件。
- 能否创建任务。
- 能否邀请其他虚拟员工。
- 能否调用某类工具。
- 能否访问某个 WorkEnvironmentNode。
- 能否向外部渠道发送消息。

虚拟员工的 prompt 不应成为权限系统。Prompt 可以提醒虚拟员工遵守规则，但最终执行必须由策略引擎和服务端校验。

## 4. 审批模型

审批应该围绕具体风险操作触发。

常见审批场景：

- 文件写入。
- 文件删除。
- 执行命令。
- 访问敏感路径。
- 访问外部网络或内网系统。
- 调用付费模型或高成本工具。
- 使用云工作环境。
- 向第三方系统提交内容。
- 邀请或通知外部成员。

审批结果可以是：

- 允许本次。
- 本工作上下文内允许。
- 本会话内允许。
- 对该工具规则长期允许。
- 拒绝。

审批需要记录：

- 谁发起。
- 哪个虚拟员工发起。
- 哪个工作上下文。
- 要执行什么。
- 风险说明。
- 谁批准或拒绝。
- 生效范围。
- 时间。

## 5. 工作上下文隔离

工作上下文用于隔离任务状态，但不是安全边界的全部。

它需要支持：

- 相关消息引用。
- 参与者列表。
- 可见范围。
- 工具结果范围。
- 交付物范围。
- 子 Agent Session 范围。

当一个工作上下文 fork 时，需要重新计算权限和资源可见性，不能简单复制所有内部状态。

## 6. WorkEnvironmentNode 沙箱

本地和云端工作环境都需要沙箱策略。

策略维度包括：

- 文件系统路径。
- 写入权限。
- 删除权限。
- 命令白名单或黑名单。
- 网络访问。
- 进程和资源限制。
- 环境变量。
- 凭据访问。
- 输出大小。
- 执行超时。

一台设备提供多个隔离空间时，每个空间都应有独立策略和审计记录。

## 7. 云工作环境隔离

云工作环境由 VT 提供，安全要求更高：

- 租户隔离。
- 节点隔离。
- 空间隔离。
- 凭据隔离。
- 网络隔离。
- 镜像和工具供应链审查。
- 操作审计。
- 数据清理和保留策略。

云工作环境不只是“远程机器”，而是产品化的受控执行环境。

## 8. 消息与可见性

消息系统需要区分：

- 用户可见消息。
- 系统消息。
- Agent 内部上下文。
- 工具执行日志。
- 审计事件。
- 交付物。

并非所有内容都应该进入聊天窗口，也并非所有内部内容都应该对所有成员可见。

特别需要避免：

- 把工具结果泄露到无权限频道。
- 把某组织的上下文注入另一个组织任务。
- 把用户私聊内容错误关联到公共工作上下文。
- 把敏感附件放入低权限虚拟员工的 prompt。

## 9. 审计

审计记录应覆盖：

- 消息触发。
- 工作上下文创建、关联、fork。
- 虚拟员工状态变化。
- VTA Session 创建、恢复、重建。
- 工具调用。
- WorkEnvironmentNode 执行。
- 审批请求和响应。
- 交付物生成和发布。
- 权限变更。
- 配置包变更。

审计不是给普通聊天界面展示的内容，但必须可查询、可追溯、可用于安全分析和企业合规。

## 10. VTA 相关安全要求

VT 对 VTA 的潜在要求包括：

- Store 层租户隔离。
- Tool call 权限中断和恢复。
- 远程工具执行抽象。
- 子 Agent parent 链路审计。
- MessageStore 与 EventStore 双轨一致性。
- Prompt 注入来源可追溯。
- 配置包版本可审计。

这些要求需要在 VTA 设计中保留扩展点。
