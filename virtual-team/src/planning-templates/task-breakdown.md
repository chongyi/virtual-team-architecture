# 任务拆解模板

> 此模板用于将史诗级功能拆解为可分配给开发人员或 AI Agent 执行的独立任务。任务拆解应遵循"单人可完成、单 Sprint 内可交付、可独立测试"的原则。

## 史诗/功能引用

| 字段 | 内容 |
|------|------|
| **功能 ID** | |
| **功能名称** | |
| **来源** | 路线图里程碑 M-XX / 技术方案文档 / 产品需求 |
| **所属轨道** | A / B / C |
| **协议边界面** | 涉及哪些协议接口 |

## 任务列表

| ID | 描述 | 类型 | 估时 | 负责人 | 依赖 | 验收条件 |
|----|------|------|------|--------|------|---------|
| T-01 | | feat/fix/refactor/test/docs | Xh | | — | AC-01 |
| T-02 | | | | | T-01 | |

### 任务类型

- `feat` — 新功能实现
- `fix` — Bug 修复
- `refactor` — 重构（无行为变化）
- `test` — 测试（单元/集成/E2E）
- `docs` — 文档

### 估时指南

- < 2h：非常明确的小改动
- 2-4h：标准任务，有清晰输入输出
- 4-8h：复杂任务，涉及多文件或多层
- > 8h：应进一步拆解

## 子任务拆解规则

对于超过 8 小时的任务，按以下维度拆解：

1. **协议先行**：先定义接口/数据结构，再实现功能
2. **Mock 验证**：在真实依赖就绪前，用 mock 实现验证接口设计
3. **数据层 → 服务层 → 接入层**：自下而上地拆解后端任务
4. **核心路径 → 边界情况**：先实现 Golden Path，再补充错误处理和边界
5. **单文件 → 单模块 → 跨模块**：按影响范围从小到大

### 拆解示例

```
功能：消息发送 API

Task 1 (4h)：定义消息请求/响应的 Rust 结构体（protocol crate）
Task 2 (2h)：在 data-model-reference.md 中定义消息表 DDL
Task 3 (4h)：实现消息持久化层（MessageStore）
Task 4 (3h)：实现消息发送 handler（HTTP + WebSocket）
Task 5 (2h)：添加幂等性（client_request_id 去重）
Task 6 (2h)：添加权限检查（频道成员验证）
Task 7 (1h)：添加可观测性（tracing span + metrics）
Task 8 (4h)：集成测试（发送 → 持久化 → 广播）
```

## 验收条件模板

每个任务应关联验收条件：

```
AC-XX: {任务ID} - {条件描述}

Given: {前置条件}
When: {触发动作}
Then: {期望结果}

边界情况:
- {边界情况1}: {期望行为1}
- {边界情况2}: {期望行为2}
```

## 就绪定义 (Definition of Ready)

以下条件全部满足，任务才可被认领：

- [ ] 输入和输出有明确规格或引用协议文档
- [ ] 依赖项已就绪（或可 mock）
- [ ] 验收条件已定义
- [ ] 估时合理（< 8h，或已拆解）
- [ ] 无阻塞项

## 完成定义 (Definition of Done)

以下条件全部满足，任务才算完成：

- [ ] 代码通过 CI（lint + test + build）
- [ ] 单元测试覆盖核心逻辑
- [ ] 涉及 API 变更时同步更新相关协议文档
- [ ] 涉及数据变更时同步更新 data-model-reference.md
- [ ] 新增的跨服务调用有 tracing span
- [ ] 错误路径有用户可理解的错误消息（如果是用户可见错误）
- [ ] PR 已审查并合入 main
