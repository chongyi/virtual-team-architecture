# 协作应用

## 定位

协作应用是 Virtual Team 的**用户入口**。它以类 Slack/飞书的企业协作应用形态呈现，是用户管理虚拟团队和与虚拟员工交互的唯一界面。

核心定位：

- **无虚拟员工时**：一个纯粹的即时通讯协作工具
- **虚拟员工接入后**：用户与虚拟员工通讯的桥梁

## 核心功能模块

### IM 通讯

即时消息是用户与虚拟员工交互的主要方式：

- 虚拟员工作为**联系人**出现在通讯录中，与真人用户身份一致
- 消息为 Append-Only 模式（聊天记录追加），类似传统 IM
- 支持文本、富文本、文件、特殊消息等多种消息类型
- 虚拟员工发送的消息与真人发送的消息在 UI 上无本质区别
- 支持群组和频道，多虚拟员工可参与群组协作

### 协作工具集

协作工具是工作产出和详细内容的载体，对应现实中企业应用的文档、表格、审批流等。虚拟员工的详细工作过程、中间产物和最终交付内容不是事无巨细地输出到聊天框，而是通过协作工具呈现：

- 文档/知识库
- 表格/多维表格
- 任务看板
- 审批流
- 自定义协作组件

协作工具集可扩展，类似 MCP Server 的机制，允许第三方开发协作插件。

### 组织管理

用户通过应用创建和管理组织架构：

- 创建/删除/修改组织
- 组织树状嵌套管理
- 虚拟员工的创建、配置、分配

### 虚拟员工接入管理

用户管理哪些虚拟员工可以接入其账户：

- 浏览和选择虚拟员工（内置市场 + 第三方）
- 配置虚拟员工的工作环境节点
- 设置虚拟员工的权限边界

## 消息的上下文增强

协作应用不只是消息的"搬运工"，它还为虚拟员工提供**上下文增强**能力，减少 token 消耗和分析复杂度。

### 消息标记

消息结构除了传统 IM 字段（内容、类型、时间戳、SeqID）外，包含扩展的标记字段用于虚拟团队场景：

- 关联的工作上下文 ID
- 消息意图分类标记
- 引用/关联消息指针

当意图识别 Agent 确认某条消息与特定工作上下文相关后，可通过上层 API 通知协作应用写入标记。后续拉取消息记录时可快速过滤关联消息，避免每次都全量分析。

### 前置上下文构建

协作应用在转发消息到虚拟员工时，会基于已有标记和 RAG 机制预先构建一个精简的上下文数据段，随消息一同发送给意图识别 Agent。这避免了 Agent 反复通过 tool call 搜索历史消息，显著减少 token 和带宽消耗。

```
消息到达 → 提取标记 → RAG 检索关联历史 → 构建上下文数据段 → 与消息一同投递
```

## 与虚拟员工系统的关系

协作应用与虚拟员工系统之间通过**协议层**对接：

- 协作应用不直接调用 VTA API
- 虚拟员工通过协议层作为一个"外部客户端"接入
- 消息收发、状态同步、事件推送均通过协议承载

协议层详见 [协议与集成](./11-protocol-and-integration.md)。
