# 项目愿景与场景拟合

## 行业现状

### LLM 的底层限制

无论 Agent 实现得多优秀，其本质仍然是基于 LLM API 之上。当前阶段 LLM 存在无法直接解决的问题，且这些问题之间相互交错：

1. **长上下文细节丢失**：对于单一对话，过长的上下文会导致 LLM 丢失诸多细节，需要通过压缩解决但无法根治。
2. **压缩也无法保证完整保留**：即使支持更长上下文或引入压缩机制，细节丢失不可避免，仅能延长对话持续能力。
3. **注意力机制不稳定**：LLM 底层仍依赖注意力机制，权重标准并非稳定可靠。对于非专业用户，模糊的提示词会进一步加重这个现象。

### 当前 Agent 方案的门槛

市场上已有的成熟 Agent 方案（Claude Code、Codex、OpenCode 等）在编码开发领域表现出色，但对于普通用户仍存在显著门槛：

1. **使用门槛高**：CLI 安装和交互是主流，即使提供图形化界面也需要大量手动配置。
2. **概念学习成本大**：MCP、Skills、Prompt、Token 等概念需要一定知识储备才能有效使用。
3. **权限与安全配置复杂**：许多安全边界取决于使用者自身判断，或在易用性和安全性之间做取舍。

### 不同 Agent 的能力差异

不同模型和 Agent 各有所长，且更新迭代节奏不同。使用者需要自行维护本地版本、管理配置——这对于非专业用户是巨大负担。

## 场景拟合

### 从公司运营到虚拟团队

现实世界中，复杂工作无法由单一个体完成，需要通过**组织协作**来实现。我们以"公司"作为拟合场景展开：

| 现实概念 | Virtual Team 映射 | 说明 |
|----------|------------------|------|
| 公司拥有者/创始人 | **用户** | 通过应用掌控整个虚拟团队 |
| 公司 | **组织** | 树状结构，可嵌套子组织 |
| 部门/工作组 | **子组织** | 服务于特定类别工作，可有主管型员工 |
| 员工 | **虚拟员工** | 具备特定能力的 AI Agent，使用工具完成工作 |
| 助理 | **助理型虚拟员工** | 协调管理、任务分派、工作汇报 |
| 部门主管 | **Leader 型虚拟员工** | 工作拆分、分派、协调、汇总、审查 |
| 办公设备/软件 | **工作环境节点** | 提供工具、MCP、沙盒等执行能力 |
| 工作台 | **工作上下文** | Agent 的独立工作会话 |

### 为什么是协作应用形态

Slack、飞书等企业协作应用已被全球用户广泛接受，从 PC 到移动端几乎没有上手门槛。Virtual Team 选择以类似形态作为用户入口，使任何用过此类应用的用户都能**零学习成本**开始使用。

### 虚拟员工：像"人"一样工作

虚拟员工不是"能聊天的 Bot"，而是：

- **有明确岗位**：不同配置包定义不同角色（销售、客服、开发）
- **使用工具**：调用文件系统、数据库、浏览器、API、外部 Agent
- **与人协作**：与其他虚拟员工或人类用户配合完成任务
- **独立工作**：有自己的工作区，不会把过程细节全部输出到聊天框

就像现实中的聊天工具，对话窗口用于沟通确认、简短回复和交付说明。详细的工作流程、中间产物和大型交付内容则通过协作工具展示。

## 设计哲学

### 配置包驱动

虚拟员工的所有能力通过**文件式配置包**注入而非硬编码：

```
my-virtual-employee/
├── manifest.toml          # 包元信息
├── prompts/
│   ├── system.hbs         # 系统提示模板
│   └── instruction.hbs    # 角色指令
├── scenes/                # 按场景路由的 prompt
├── providers/             # 模型特化覆盖（可选）
└── skills/                # 技能声明
```

配置包是文件资产，可版本控制、代码审查、CI/CD 部署。虚拟员工的"角色定义"本质上是代码资产，而非运行时可随意篡改的数据库状态。

### 分层开放

Virtual Team 支持用户引入外部能力——第三方 Agent、MCP Server、自定义工具——但所有这些都通过虚拟员工内部的工作环境节点进行，保持清晰的归属和权限边界。
