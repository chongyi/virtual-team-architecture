# 协作应用技术方案总览

## 定位

本文是协作应用的技术方案冻结入口，用于把协作应用从产品概念推进到可实施的工程方案。它不替代[协作应用总览](../overview.md)，而是在产品边界已经明确后，冻结基础版的客户端、服务端、协议、数据、权限、同步可靠性与可观测性口径。

协作应用必须作为独立系统成立。Agent Server、虚拟员工系统和工作环境节点都是接入方或下游系统；它们不可成为协作应用启动、登录、IM、组织、基础协作工具和搜索能力的硬依赖。

## 基础版实施边界

基础版要求做到“可独立上线、可持续扩展、可接入虚拟员工”，而不是一次性实现完整协作套件。

必须冻结并实施：

- 用户认证、租户切换、组织和成员管理。
- 频道、群组、私聊、消息追加、线程回复、反应、编辑、软删除、在线状态和多端同步。
- WebSocket 实时事件与 HTTPS REST 历史查询。
- 文件上传、附件引用、对象存储元数据和权限检查。
- 协作对象通用对象壳，以及文档、表格、看板、审批、日程/定时器五类第一方内置扩展的基础版。
- Tool Registry、Extension Manifest、Tool Action Gateway、权限、审计、搜索索引调度、通知聚合和 IM 引用。
- Agent Server 接入预留点：虚拟员工成员类型、消息转发、markers 回写、context segment、Tool Action 调用和主动通知。
- 独立管理端的 Admin API、Admin RBAC、Admin Audit 和高风险操作审批边界。

基础版不承诺：

- 文档实时多人协同编辑引擎。
- 多维表格公式引擎、多视图、复杂关联记录和仪表盘。
- 第三方插件市场。
- 客户端运行时动态加载任意 Flutter/Dart 插件。
- 具体开源项目选型，如文档编辑器、表格引擎或 CRDT/OT 框架。

## 核心原则

### 独立应用优先

协作应用是通讯协作产品，不是 Agent Server 的附属 UI。Agent Server 不可用时：

- 用户仍可登录和切换租户。
- IM 消息、频道和历史查询仍可工作。
- 第一方协作工具仍可由用户手动创建、查看和编辑。
- 与虚拟员工有关的入口显示离线、排队或不可用状态，但不阻塞协作应用核心能力。

### 核心与扩展分离

协作应用核心只管理通用平台能力：对象壳、权限、审计、搜索、通知、IM 引用、租户隔离和配额。具体工具的数据模型和操作语义由第一方内置扩展提供。

所有工具操作都必须经过 Tool Action Gateway。用户 UI、系统任务和 VE 都不能绕过核心权限、审批、审计和配额路径直接修改扩展数据。

### 基础版可实施，完整形态可演进

文档和多维表格的完整形态复杂度较高，因此基础版只冻结轻量结构化内容、乐观并发、类型化数据表、基础筛选排序和导出能力。实时协同、公式、多视图和复杂嵌入作为完整形态方向记录，不作为首期实现承诺。

### 多端共享业务，平台能力适配

Flutter 客户端采用单代码库。协议、模型、状态管理、缓存、同步和权限逻辑默认共享；布局、通知、文件、窗口、深链和系统权限通过平台能力适配层处理。

Mobile、Desktop、Web 的网络、存储、推送和发布策略不能混同。移动端以前台实时和后台推送+补拉为主；桌面端可承担更大的缓存和本地搜索；Web 端受浏览器沙盒、存储配额和通知权限约束。

### 管理端独立治理

管理端是平台内部全后台，不是普通协作应用客户端的一组隐藏页面。它使用独立 Web 应用、独立 Admin API、独立权限域和独立审计，覆盖租户、用户、VE、计费、配额、扩展、风控、客服、系统任务和数据分析。

### 可观测性进入基础设计

每个跨服务请求、消息投递、扩展动作、搜索索引任务和通知投递都必须携带 request id 或 correlation id。审计日志用于回答“谁在什么租户、以什么身份、对哪个对象做了什么操作”，业务日志和指标用于定位系统健康问题。

## 冻结验收口径

技术方案达到冻结状态时，实施团队应能明确回答：

- 客户端每个端共用哪些逻辑，哪些能力必须平台适配。
- 服务端每个模块的职责、输入输出和失败边界。
- 消息、频道、对象、扩展动作和 Agent 接入分别走什么协议。
- 哪些数据属于协作应用核心，哪些数据属于工具扩展或 Agent Server。
- 用户、VE 和系统任务分别如何通过权限、审批、审计和配额检查。
- 断线、重复请求、服务降级、搜索延迟和通知失败时用户会看到什么行为。
