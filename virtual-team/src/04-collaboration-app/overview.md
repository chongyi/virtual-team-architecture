# 协作应用

## 定位

协作应用是 Virtual Team 的**用户入口**，也是一个可以独立运行的即时通讯与协作系统。它以类 Slack/飞书的企业协作应用形态呈现，是用户管理虚拟团队、组织沟通、沉淀工作产物以及与虚拟员工交互的唯一界面。

核心定位：

- **无虚拟员工时**：一个纯粹的即时通讯协作工具
- **虚拟员工接入后**：虚拟员工作为特殊联系人接入协作网络，用户通过同一套消息、频道、协作工具与其协作

关键边界：不是协作应用接入虚拟员工系统，而是虚拟员工系统以外部客户端/外部服务的方式接入协作应用。协作应用不依赖 Agent Server 才能成立；Agent Server 不可用时，协作应用仍应保持 IM、组织、协作工具等基础能力可用。

## 技术栈

| 层 | 技术 | 理由 |
|---|------|------|
| **客户端** | Flutter 3.x | 跨平台（iOS/Android/Desktop/Web），单代码库，富 UI 表现力 |
| **管理端** | Vite + React + React Router 7 + Tailwind CSS + shadcn/ui + Zustand | 独立 Web 管理端，适合密集后台、运营、客服、风控和系统管理 |
| **服务端** | Rust (tokio, axum) | 高性能、内存安全、与 VTA 技术栈一致 |

## 与虚拟员工系统的关系

协作应用与虚拟员工系统之间通过**协议层**对接：

```
用户客户端 ←→ 协作应用服务端 ←→ 协议层 ←→ Agent 服务器 ←→ 虚拟员工
```

协作应用本身是**独立系统**，即使未挂载虚拟员工系统，仍然可以作为纯粹的即时通讯协作工具运行。虚拟员工作为"联系人"通过协议接入——与用户通过客户端接入的方式对等。

协议层详见 [协议与集成](../11-protocol-and-integration/overview.md)。

## 独立运行边界

协作应用在不依赖虚拟员工系统的情况下，必须独立提供：

- 用户认证、Tenant 切换和基础权限。
- 组织、频道、群组、私聊和成员管理。
- WebSocket 实时通讯与 REST 历史查询。
- Append-only 消息持久化、编辑、软删除、反应和线程回复。
- 文件上传、对象存储引用和消息附件。
- 协作工具的基础 CRUD 和权限检查。
- 搜索、在线状态、多端同步和离线重连。

虚拟员工相关能力在协作应用中表现为扩展接入点，而不是核心依赖：

- `sender.type = virtual_employee` 和频道成员中的虚拟员工成员类型。
- 消息 `markers` 字段。
- `context_segment` 构建与转发。
- Agent Server 对接适配器。
- VE 管理界面、在线状态和工作上下文视图。
- 审批卡片、工作摘要卡片等 Virtual Team 特化消息类型。

管理端是平台内部运营和治理系统，不属于普通协作应用客户端。管理端以独立 Web 应用形式存在，通过 Admin API 管理租户、用户、VE、计费、资源、风控、审计、工单和系统任务。

## 冻结口径

协作应用方案要达到可冻结状态，至少需要同时冻结以下内容：

- 产品边界：协作应用独立运行时提供哪些能力，虚拟员工接入后新增哪些能力。
- 协议边界：客户端协议、对接协议、Agent Server 回写 API 的请求/响应和错误语义。
- 数据边界：IM、组织、协作工具、markers、context segment、搜索索引的权威数据归属。
- 状态边界：消息、频道、成员、在线状态、协作工具对象、Schedule/Timer 的状态机。
- 降级边界：Agent Server、工作环境节点、RAG、搜索、对象存储不可用时的用户感知和系统行为。
- 验收边界：每个模块的最小可测试场景、幂等规则、权限规则和可观测性指标。
