# Agent 服务器

## 定位

Agent 服务器是虚拟员工系统在**服务端**的承载。它独立于协作应用，负责接收消息、管理虚拟员工生命周期、调度 Agent 执行。

## 组成

```
Agent 服务器
├── 接入层
│   ├── 协议适配（对接协作应用）
│   ├── 消息收发
│   └── 事件推送
└── 虚拟员工管理服务
    ├── 虚拟员工生命周期管理
    ├── 虚拟员工实例路由
    ├── 多租户调度
    └── 工作环境节点调度
```

## 接入层

### 职责

接入层是协作应用与虚拟员工系统之间的**协议桥梁**：

- 接收协作应用转发的消息，解析并路由
- 将虚拟员工的回复格式化为协作应用可消费的格式
- 处理虚拟员工的主动通知（工作完成、需要确认等）
- 协议转换——协作应用使用应用层协议，虚拟员工系统内部使用 VTA 协议

### 协议适配

接入层屏蔽协作应用协议与内部协议的差异：

```
协作应用协议 ←→ 接入层 ←→ 内部协议 (VTA / JSON-RPC)
```

这使得协作应用和虚拟员工系统可以独立演进。协作应用的协议变化只需更新接入层的适配逻辑，不影响虚拟员工内部。

## 虚拟员工管理服务

### 定位类比

虚拟员工管理服务类比于**人力资源公司**——一个人服务多家公司，由人力资源公司统一调度。

在 Virtual Team 中：一个虚拟员工管理服务管理多个虚拟员工实例，这些实例服务于不同租户（用户）。

### 职责

**虚拟员工生命周期管理**：

- 创建：根据用户选择的配置包初始化虚拟员工实例
- 挂载：将虚拟员工激活并注册到消息路由表
- 挂起：不活跃实例持久化状态后释放内存资源
- 恢复：从持久化状态恢复实例
- 卸载/销毁：移除虚拟员工并回收所有资源

**虚拟员工实例路由**：

- 维护消息到虚拟员工的映射表
- 根据消息目标定位正确的虚拟员工实例
- 处理群组消息的多目标路由

**多租户调度**：

- 租户 = 用户级别。每个用户的数据空间完全隔离
- 消息路由前验证租户归属
- 确保跨租户无数据泄露

**工作环境节点调度**：

- 管理用户所有工作环境节点的在线状态
- 将虚拟员工的工具调用请求转发到正确的工作环境节点
- 处理节点离线/重连时的容错

## 扩容设计

### 无状态管理服务

虚拟员工管理服务设计为**无状态**：

- 虚拟员工的状态持久化在 Store 中（数据库/消息队列）
- 管理服务实例可以水平扩展
- 任意管理服务实例可以接管任意虚拟员工实例

### 虚拟员工实例分布

虚拟员工实例可分布在不同的进程/节点上：

```
                    ┌──────────────────┐
                    │   负载均衡/路由   │
                    └────────┬─────────┘
           ┌─────────────────┼─────────────────┐
           ↓                 ↓                   ↓
    ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
    │ Agent Node 1 │  │ Agent Node 2 │  │ Agent Node N │
    │ VE-1 VE-3    │  │ VE-2 VE-5    │  │ VE-4 VE-6    │
    └──────────────┘  └──────────────┘  └──────────────┘
```

### 冷热分离

- **热实例**：活跃（有未完成工作或近期有消息交互）的虚拟员工，常驻内存
- **冷实例**：不活跃的虚拟员工，状态持久化后释放内存，恢复时从持久化重建
- 冷热切换由管理服务根据消息到达和活跃时间自动决策

## 与 VTA 的关系

Agent 服务器的虚拟员工管理服务是基于 VTA 构建的上层服务：

- 不直接操作 VTA Store/Message/Event
- 通过 VTA Host 的 API 创建和管理 Session
- 虚拟员工的配置包、场景路由、模型选择等能力由 VTA Runtime 承载

Agent 服务器关注的不是"Agent 如何推理"，而是"虚拟员工如何被管理、消息如何被路由、资源如何被调度"——这些是高于单个 Agent Runtime 的编排层问题。
