# 租户与组织模型

## 租户模型

### 定义

租户（Tenant）= **用户**。一个用户 = 一个独立的 Virtual Team 数据空间。

### 租户的隔离范围

| 隔离项 | 说明 |
|--------|------|
| 虚拟员工实例 | 每个用户的虚拟员工完全独立，不可跨用户访问 |
| 消息与对话 | 用户间消息完全隔离 |
| 工作上下文 | 工作上下文归属创建它的用户 |
| 组织数据 | 用户创建的组织结构仅自己可见 |
| 配置包 | 用户导入的配置包独立存储 |
| 工作环境节点 | 用户拥有的节点仅自己可使用 |

### 数据模型预留

在数据模型层面，所有核心实体预留 `tenant_id` 字段，Store 层默认带租户过滤。

## 组织模型

### 定义

组织是用户创建的逻辑管理单元，呈**树状结构**：

```
用户
└── 组织 A（公司）
    ├── 组织 A-1（部门）
    │   ├── 虚拟员工 1
    │   └── 虚拟员工 2
    ├── 组织 A-2（部门）
    │   └── 虚拟员工 3
    └── 虚拟员工 4（直属员工）
```

### 组织的用途

**资源与数据划分**：
- 不同业务的虚拟员工归属于不同组织
- 组织级别的配置和策略管理

**工作路由辅助**：
- 助理接收到任务后，分析任务类型匹配到合适的组织
- 组织内的 Leader 型虚拟员工负责进一步分派

**减少 token 消耗**：
- 组织有元数据和描述，便于 AI 快速识别匹配
- 不需要每次分析任务都扫描所有虚拟员工

### 组织中的角色

| 角色 | 说明 |
|------|------|
| **Leader** | 组织主管型虚拟员工，负责工作拆分、分派、协调 |
| **Member** | 普通成员型虚拟员工，具体执行工作 |
| **Assistant** | 助理型虚拟员工，协调管理，不属于特定组织 |

### 用户感知程度

在大多数简单场景下，组织对用户是透明的。只有在业务复杂度增加到需要分层管理时才显式创建——类似现实中个体商户 vs 企业集团。

## 组织与虚拟员工调度的关系

### 任务路由

```
用户发消息 → 助理分析
              ├── 判断任务类型
              ├── 搜索匹配的组织（基于组织元数据）
              ├── 找到匹配组织 → 交给该组织的 Leader
              │                    └── Leader 决定自己完成还是分派
              └── 未找到匹配 → 直接匹配具体的虚拟员工或与用户确认
```

### 组织的元数据

每个组织有结构化的元数据，便于 AI 分析匹配：

- 组织名称和描述
- 业务领域标签
- 成员虚拟员工列表及其能力摘要
- 典型任务类型

这些元数据由系统自动维护——创建虚拟员工时自动更新，无需用户手动填写。
