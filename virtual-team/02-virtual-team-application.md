# 02. 虚拟团队应用

## 1. 定位

虚拟团队应用首先是一个通讯协作工具。即使没有虚拟员工接入，它也应该能够作为 IM 和轻量协作平台独立成立。

当 AgentServer 接入后，虚拟员工会作为特殊成员进入这个协作网络，接收消息、发出回复、参与群组、处理任务、提交交付物。

## 2. 基础通讯形态

虚拟团队应用需要支持多种消息传播方式：

- 一对一私聊：用户与用户、用户与虚拟员工、虚拟员工与虚拟员工。
- 群聊：多个成员围绕一个临时事项沟通。
- 频道：组织内围绕主题、部门、项目、流程形成长期空间。
- 跨组织协作空间：在权限允许时支持多组织成员协同。
- 外部渠道接入：第三方 IM、协作软件、邮件、Webhook 等。

这些形态对 Agent 的本质影响是消息上下文不同，而不是 Agent 类型不同。VTE 需要知道：

- 消息从哪里来。
- 谁发的。
- 发给谁。
- 在什么组织、频道、群组或私聊里。
- 是否提及自己。
- 是否引用了已有消息、任务或交付物。

## 3. 协作应用能力

产品层需要提供以下基础能力：

- 组织管理。
- 成员目录。
- 私聊、群聊、频道。
- 消息发送、编辑限制、撤回策略、引用、回复、提及。
- 文件和附件。
- 任务视图。
- 交付物视图。
- 通知与待办。
- 审批入口。
- 权限与角色管理。
- 虚拟员工目录。
- 工作环境节点管理。

这些能力中，消息系统是入口，但不是全部。复杂任务的状态、交付和审计需要进入协作工具中的专门视图。

## 4. 虚拟员工接入前后

### 4.1 未接入 AgentServer

应用表现为普通通讯协作工具：

- 人类用户之间沟通。
- 组织和频道管理。
- 文件、任务、通知等基础协作。
- 不存在虚拟员工在线、接收消息或执行任务。

### 4.2 接入 AgentServer

AgentServer 挂载后，虚拟员工可以作为成员出现：

- 在成员目录中可见。
- 可被邀请进入组织、频道或群组。
- 有在线、离线、忙碌、等待确认等状态。
- 可以接收消息并回复。
- 可以创建和维护工作上下文。
- 可以调用应用 API 标记消息、更新任务、提交交付物。
- 可以根据权限请求使用工作环境节点。

可以把虚拟员工理解为一个需要注册登录并适配协议的成员。它如果没有接入应用协议，就无法正确接收消息、理解消息形态和操作协作工具。

## 5. 私人小助理

用户首次进入系统时，默认提供一个私人小助理。

私人小助理是一个 VTE，核心职责包括：

- 引导用户创建组织。
- 解释虚拟团队的基础使用方式。
- 协助招募或创建虚拟员工。
- 建议组织结构、频道和工作环境配置。
- 帮助用户把模糊需求转化为可执行任务。
- 在需要时协调其他虚拟员工。

私人小助理的商业化升级不改变其底层身份。基础助理和高级助理在核心代码上仍属于同一套虚拟员工 Agent 体系，只是模型、工具、权限、配置包、专属服务端能力和协作策略不同。

## 6. 用户使用路径

典型首次使用路径：

1. 用户注册并进入虚拟团队应用。
2. 系统自动创建私人小助理。
3. 私人小助理引导用户创建第一个组织。
4. 用户描述自己的业务或目标。
5. 私人小助理建议可招募的虚拟员工岗位。
6. 用户选择或定制虚拟员工。
7. 系统根据岗位配置 VTE、模型策略、工具需求和权限策略。
8. 如果虚拟员工需要本地或云端工具，系统引导用户配置 WorkEnvironmentNode。
9. 虚拟员工进入组织、频道或私聊，开始参与协作。

## 7. 聊天体验规则

聊天框中的表现应贴近真实协作：

- 简单明确的问题直接回复。
- 不明确的需求通过追问确认。
- 明确的新工作目标自动创建或关联工作上下文。
- 工作过程中遇到问题时回到聊天中确认。
- 重要阶段通过聊天汇报进展。
- 完成后通过聊天说明结果，并链接交付物或任务视图。

系统不应把所有工具执行步骤都塞进聊天框。用户需要的是可理解的沟通节奏，而不是 Agent 内部执行流刷屏。

## 8. 应用 API 与虚拟员工

虚拟员工需要使用应用 API 完成消息前处理和后处理：

- 接收消息。
- 发送消息。
- 标记消息已读或已处理。
- 关联消息到工作上下文。
- 创建任务或更新任务状态。
- 上传或链接交付物。
- 发起审批请求。
- 查询组织、频道、成员和权限上下文。

这些应用 API 是 VTE 适配虚拟团队应用的关键。VTE 的核心工作逻辑可以与应用协议解耦，但消息接入、上下文标记和协作工具操作必须通过应用协议完成。
