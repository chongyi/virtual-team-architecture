# 01. 概念模型

## 1. 核心概念表

| 概念 | 英文暂名 | 含义 |
|------|----------|------|
| 虚拟团队 | Virtual Team / VT | 整个项目与产品系统 |
| 虚拟团队应用 | Virtual Team Application | 类 Slack、飞书的 IM 与协作应用 |
| 租户 | Tenant | SaaS 隔离单位，通常对应一个客户账号、企业或独立使用空间 |
| 用户 | User | 系统使用者，可以创建组织、管理虚拟员工和发起任务 |
| 组织 | Organization | 用户理解团队结构的核心模型，可表示公司、部门、工作组、项目组 |
| 成员 | Member | 组织内可参与沟通和协作的主体，包含用户、人类成员和虚拟员工 |
| 虚拟员工 | Virtual Employee | 面向用户的 AI 员工概念，是 VTE 的产品表现 |
| VTE | Virtual Team Employee-agent | 虚拟员工 Agent，高阶 Agent 封装 |
| 私人小助理 | Personal Assistant | 默认创建的虚拟员工，负责引导、组织管理和基础协调 |
| AgentServer | AgentServer | 服务端管理虚拟员工 Agent 的服务，名称暂定 |
| 消息网关 | Message Gateway | 应用消息和外部渠道消息进入 AgentServer 的路由层 |
| 工作上下文 | Work Context | 围绕任务目标的数据层上下文切片 |
| 工作环境节点 | WorkEnvironmentNode | 虚拟员工可使用的本地或云端工具执行环境 |
| VTA | Virtual Teams Agent | Pure Agent Runtime，支撑 VTE 执行 |

## 2. 现实组织映射

VT 使用现实组织模型降低用户理解成本。

| 现实场景 | VT 概念 | 说明 |
|----------|---------|------|
| 公司、集团、部门、项目组 | Organization | 统一用组织表达，可形成树状结构 |
| 雇主、负责人 | User | 用户掌控虚拟组织和虚拟员工 |
| 员工 | Virtual Employee / VTE | 具备岗位、职责、工具和工作上下文 |
| 助理、人事、行政 | Personal Assistant / 管理型 VTE | 负责引导、协调、建议、招聘、组织管理 |
| 主管、负责人 | 管理型虚拟员工 | 可拆分、分派、检查和汇总任务 |
| 办公设备、软件、内部系统 | WorkEnvironmentNode / Tools | 虚拟员工完成工作的工具和环境 |
| 聊天、会议、邮件 | Message / Channel / Conversation | 沟通和协作入口 |
| 任务、事项、项目 | Work Context / Task View | 任务目标和执行状态的上下文容器 |

## 3. 组织模型

组织是用户视角下的现实模型，也是虚拟员工理解上下文的重要信息来源。

组织可以表达：

- 独立公司。
- 子公司。
- 部门。
- 工作组。
- 项目组。
- 临时协作空间。

组织模型应支持树状结构，但不把“部门”和“公司”设计成完全不同的底层实体。它们本质上都是组织节点，只是在产品展示和权限策略中具有不同语义。

### 3.1 用户视角

用户通过组织理解和管理虚拟团队：

- 当前有哪些组织。
- 每个组织有哪些成员。
- 哪些虚拟员工属于或服务于该组织。
- 哪些频道、群组、任务和交付物归属于该组织。
- 哪些权限、工具和工作环境可以被该组织使用。

### 3.2 虚拟员工视角

在虚拟员工底层视角，组织主要是消息和任务中的上下文要素：

- 消息来自哪个组织。
- 当前对话属于哪个组织或频道。
- 发起人和参与者在组织中的身份。
- 任务需要遵守该组织的规则、偏好和权限。

这些信息会进入 VTE 的 prompt 和动态上下文，帮助虚拟员工做出符合组织场景的判断。

### 3.3 跨组织复用

虚拟员工可以跨组织复用。系统不应限制一个虚拟员工只能属于一个组织。

跨组织复用时，需要明确：

- 当前消息和任务属于哪个组织。
- 当前组织允许该虚拟员工访问哪些资源。
- 虚拟员工在不同组织中的角色、权限和记忆边界。
- 长期记忆哪些是员工级共享，哪些是组织级隔离。

## 4. 虚拟员工模型

虚拟员工是面向用户的产品概念。VTE 是其 Agent 实现形态。

一个虚拟员工至少包含：

- 身份：姓名、头像、岗位、介绍。
- 职责：擅长什么、不负责什么。
- 组织关系：可服务哪些组织、频道和用户。
- 能力配置：模型策略、prompt 配置包、技能、服务端工具。
- 工具环境：可访问哪些 WorkEnvironmentNode。
- 权限策略：哪些操作允许、哪些需要审批、哪些禁止。
- 记忆与偏好：员工级、组织级、用户级长期上下文。
- 工作上下文：正在处理和历史处理过的任务切片。

私人小助理也是虚拟员工，只是默认创建，并且岗位偏向引导、管理和协调。

## 5. 消息模型

虚拟团队应用中的消息是 append-only 的协作记录。消息的主要作用是：

- 沟通。
- 确认。
- 提问。
- 汇报。
- 交付说明。
- 引用已有任务或上下文。

消息不应承担展示完整 Agent 内部执行日志的责任。工具调用、推理过程、中间状态和审计记录应进入 AgentServer、工作上下文、事件日志和交付物系统。

消息需要携带足够的路由与上下文信息：

- 发送者。
- 接收者。
- 所在组织。
- 所在频道、群组或私聊。
- 消息类型。
- 引用、回复、附件、提及。
- 是否已关联工作上下文。

## 6. 工作上下文模型

工作上下文是数据层的任务切片，用于解决聊天窗口和实际工作目标之间的错位。

一个聊天窗口可以同时出现多个工作事项；一个工作事项也可能跨多个消息、多个频道、多个参与者持续推进。

工作上下文用于维护：

- 任务目标。
- 已确认需求。
- 相关消息引用。
- 参与者。
- 所属组织。
- VTE 子 Agent Session。
- 工具执行记录。
- 交付物。
- 当前状态。
- 上下文摘要。
- 与其他工作上下文的 fork 或关联关系。

创建工作上下文本身是低风险的数据操作，不需要审批。需要审批的是后续涉及权限、外部访问、文件修改、命令执行、费用消耗等具体操作。

## 7. 工作环境节点模型

WorkEnvironmentNode 是虚拟员工的工具执行载体，不是 VTA。

它可以是：

- 用户本地设备上的工作环境客户端。
- 同一设备中隔离出的多个工作空间。
- 用户自有服务器上的工作环境客户端。
- VT 提供的云工作环境。

它可以承载：

- MCP Server。
- Built-in Tools。
- 浏览器自动化。
- 文件系统访问。
- 命令执行。
- 第三方成熟 Agent。
- 内部系统连接器。

虚拟员工不直接拥有设备，而是通过 AgentServer 和协议使用被分配给自己的工作环境节点。

## 8. VTA 模型

VTA 是 Pure Agent Runtime，位于 VTE 底层。

它负责让一个 Agent Session 能够被创建、运行、恢复和重建。但它不直接理解 VT 产品层的完整组织、频道、商业套餐和工作环境分配规则。

VT 需要把产品层信息转换为 VTE 配置、prompt、上下文、工具和运行时策略，再交给 VTA 执行。
