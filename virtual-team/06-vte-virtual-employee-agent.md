# 06. VTE：虚拟员工 Agent

## 1. 定位

VTE（Virtual Team Employee-agent）是虚拟员工的 Agent 实现形态。

它不是一条裸 VTA Session，而是 VTA 之上的高阶封装。VTE 要像一个组织成员一样理解消息、上下文、岗位、工具、权限和协作方式。

## 2. VTE 的组成

一个 VTE 至少由以下部分组成：

- Employee Profile：岗位、身份、职责和沟通风格。
- Prompt Package：基于 VTA PromptManager 的配置包。
- Model Policy：主模型、小快模型、场景模型和成本策略。
- Tool Policy：服务端工具、工作环境工具和审批策略。
- Message Adapter：适配虚拟团队应用消息协议。
- Work Context Manager：识别、创建、关联和 fork 工作上下文。
- Memory Manager：管理用户级、组织级、员工级长期记忆。
- Child Agent Orchestrator：创建和协调子 Agent。
- Response Composer：把内部执行结果转化为用户可见沟通。

## 3. 消息前处理与后处理

VTE 的消息处理可以分为两层。

### 3.1 前处理

前处理与虚拟团队应用协议相关：

- 接收标准化消息。
- 查询组织、频道、成员和权限上下文。
- 判断消息是否触发自己。
- 标记消息已接收或处理中。
- 建立或更新消息与工作上下文的关联。
- 必要时发起追问或确认。

### 3.2 后处理

后处理是 VTE 自身的工作逻辑：

- 根据岗位职责理解任务。
- 构建或恢复工作上下文。
- 调用 VTA Session 执行。
- 调度子 Agent。
- 使用工具。
- 生成交付物。
- 形成回复、汇报、审批请求或任务更新。

这两层要解耦。这样 VTE 的核心能力可以复用到不同消息渠道，而应用适配只需实现协议层。

## 4. 行为模式

VTE 收到消息后通常进入以下行为之一：

| 行为 | 说明 |
|------|------|
| 直接回复 | 简单、明确、无需任务上下文的问题 |
| 追问确认 | 需求不完整或有歧义 |
| 创建工作上下文 | 明确的新工作目标 |
| 关联工作上下文 | 用户补充已有任务信息 |
| Fork 工作上下文 | 基于已有任务产生新的目标 |
| 请求工具配置 | 缺少必要工具或工作环境节点 |
| 请求审批 | 操作涉及权限、风险或费用 |
| 执行并汇报 | 有足够信息和权限，进入工作执行 |

## 5. 私人小助理

私人小助理是 VTE 的一种默认岗位形态。

基础助理和高级助理在底层代码上不应分裂成两套系统。差异应来自：

- 模型配置。
- Prompt 配置包。
- 可用服务端工具。
- 组织管理能力。
- 招募和配置建议能力。
- 长期记忆容量。
- 跨员工协调能力。
- 商业化功能权限。

高级助理本质上仍是虚拟员工，只是拥有更强的能力配置和更高级的协作策略。

## 6. VTE 与组织

VTE 可以服务多个组织。

每次处理消息时，VTE 都需要根据当前消息上下文确定：

- 当前组织。
- 当前频道或群组。
- 当前发起人。
- 当前参与者。
- 当前权限边界。
- 当前组织偏好。

组织不是 VTE 的唯一归属，而是 VTE 工作时必须理解和遵守的上下文。

## 7. VTE 与工作上下文

VTE 需要把聊天中的多事项、多线程沟通转化为清晰的工作上下文。

它需要维护：

- 当前消息是否属于已有任务。
- 当前任务是否需要拆分。
- 是否需要 fork。
- 哪些历史消息对当前任务重要。
- 哪些交付物和工具结果需要保留。
- 当前任务的下一步是什么。

这层能力是虚拟员工区别于普通 Chatbot 的关键。

工作上下文并不要求由 VTE 主 Session 亲自完成。更合理的模型是：VTE 像一个岗位员工或主管，负责理解消息和协调任务；具体工作可以由该工作上下文下的独立 Agent Session 完成。这样可以让不同任务拥有独立上下文，也方便 fork、恢复、并行和审计。

## 8. VTE 与工具

VTE 可以使用两类工具：

- AgentServer 提供的服务端工具。
- WorkEnvironmentNode 提供的工作环境工具。

当没有工作环境节点时，VTE 不应直接失败，而应根据任务判断：

- 如果服务端工具足够，则继续工作。
- 如果缺少必要工具，则向用户说明需要配置哪些工具或工作环境。
- 如果任务可以改为规划、建议或文本分析，则提供替代方案。

工具缺失的处理应通过 VTE prompt 和策略实现，使虚拟员工像现实员工一样提出办公条件需求。

## 9. 子 Agent

VTE 内部可以派生子 Agent 完成独立工作。

子 Agent 适合：

- 大任务拆分。
- 并行资料检索。
- 多方案比较。
- 专项审查。
- 调用成熟第三方 Agent。

子 Agent 仍应通过 VTA 的独立 Session + parent 链接模型实现。VTE 负责把子 Agent 的结果汇总成用户可理解的工作进展和交付内容。

## 10. 用户可见输出

VTE 的用户可见输出应该像同事沟通，而不是像日志系统：

- 简洁说明当前理解。
- 必要时列出待确认问题。
- 汇报关键进展。
- 说明阻塞原因。
- 请求授权或工具配置。
- 完成后说明交付内容和后续建议。

内部推理、工具调用和临时状态默认不直接暴露，除非用户需要审计或打开详细视图。
