# 00. 项目定位

## 1. 背景

当前 Agent 产品和 Agent 框架已经覆盖了许多真实工作场景，尤其在编码、资料整理、自动化操作、浏览器任务、工作流执行等领域持续进化。但对普通用户而言，主流方案仍然存在几个明显门槛：

- 需要理解 Agent、Tool、MCP、Skill、Runtime、Provider 等概念。
- 需要安装 CLI、配置本地环境、维护模型和工具版本。
- 需要写出足够清晰的 prompt，才能获得稳定结果。
- 复杂任务往往挤在一个聊天会话里，容易出现上下文混乱和细节丢失。
- 权限、安全和本地环境访问的边界经常落到用户自己判断。

这些问题并不只来自某个产品设计，而是来自当前 LLM 和 Agent 的共同基础限制：上下文有限、注意力不稳定、工具调用需要权限、长期工作需要状态管理，且用户的表达质量会显著影响最终结果。

VT 的核心判断是：普通用户不应该先学习 Agent 工程，再使用 Agent 完成现实工作。产品应当选择一个现实中已经被广泛理解的协作模型，把底层复杂能力收敛到熟悉的组织、员工、消息、任务、工具和审批之中。

## 2. 一句话定位

Virtual Team 是一个面向普通用户和组织场景的虚拟团队系统，以协作应用作为入口，让用户像管理现实团队一样管理虚拟员工完成复杂工作。

## 3. 产品基底

VT 的产品入口是一个类似 Slack、飞书的协作应用。单独看，它首先是一个通讯协作工具：

- 用户可以创建组织。
- 用户可以建立成员关系、频道、群组和私聊。
- 用户可以发送消息、上传文件、提及成员、沉淀任务和交付物。
- 用户可以通过应用内工具完成协作。

当 AgentServer 接入后，虚拟员工会像真实成员一样注册、登录、接收消息、回复消息、标记上下文、创建工作任务并使用工具完成工作。

因此，VT 不是“聊天机器人套壳”，而是“协作应用 + 虚拟员工劳动力 + 工作环境 + Agent Runtime”的组合系统。

## 4. 顶层目标

VT 的目标不是让用户学习如何调度 Agent，而是让用户自然地完成以下事情：

- 创建一个虚拟组织。
- 通过默认私人小助理完成引导、配置和招聘。
- 招募或创建不同岗位的虚拟员工。
- 把虚拟员工加入组织、频道、群组或私聊。
- 像对人类同事一样发出需求、确认细节、查看进度、接收交付。
- 在需要时为虚拟员工配置工作环境、工具和权限。
- 在复杂任务中让多个虚拟员工协作，而不是让用户手动管理多个 Agent 会话。

## 5. 设计原则

### 5.1 自上而下

VT 的规划从真实用户、组织协作和产品使用方式出发，再逐层落到服务端、AgentServer、VTE、WorkEnvironmentNode 和 VTA。

VTA 是底层能力之一，而不是产品的起点。

### 5.2 协作应用先成立

虚拟团队应用在没有虚拟员工时，也应当是一个成立的 IM 和协作工具。虚拟员工接入后，它成为人与虚拟员工、虚拟员工与虚拟员工之间的工作入口。

### 5.3 虚拟员工是高阶 Agent

虚拟员工不是一条裸 VTA Session。它是 VTA 之上的高阶 Agent 封装，具备岗位、组织上下文、消息适配、工作上下文管理、工具环境感知、长期记忆和子 Agent 编排能力。

### 5.4 消息不是内部执行日志

聊天消息是沟通、确认、汇报和交付入口，不承载所有内部推理、工具调用和执行细节。Agent 内部执行流应当沉淀在工作上下文、审计事件、工具记录和交付物中。

### 5.5 工作上下文是数据层切片

工作上下文用于隔离任务目标、上下文、Agent 子会话和执行状态。它可以映射到用户可见的任务、事项或交付记录，但不等同于聊天窗口本身。

### 5.6 工作环境节点独立于 VTA

WorkEnvironmentNode 是工具和执行环境的承载者，可以运行在用户本地设备、隔离空间或云端主机上。VTA 是 Agent Runtime。两者职责不同，通过协议协作。

### 5.7 完整目标优先

本文档描述的是完整系统目标，而不是先以简化 MVP 限制模型。实施可以分阶段，但目标模型需要完整、可扩展、可长期演化。

## 6. 非目标

VT 不以以下方向作为核心定位：

- 只做一个通用 Chatbot。
- 只做某个 Agent CLI 的图形界面。
- 只做单用户单会话的 Agent 管理器。
- 只做工具市场或 MCP 管理器。
- 只做工作流自动化平台。

这些能力都可能成为 VT 的组成部分，但不是 VT 的最终产品形态。

## 7. 与 VTA 的关系

VTA 是 VT 的底层 Agent Runtime 基座，负责 Pure Agent 运行时能力，包括：

- LLM 推理循环。
- Prompt 配置包。
- 工具调用。
- MCP 接入。
- 消息工作轨。
- 上下文压缩。
- 子 Agent Session。
- 状态、审计和恢复。

VT 在 VTA 之上定义产品级结构：

- 用户与组织。
- 虚拟员工。
- 消息网关。
- 工作上下文。
- 工作环境节点。
- 权限与商业化。

换句话说，VTA 让一个 Agent 能可靠运行，VT 让多个虚拟员工能在真实协作场景中被普通用户理解、管理和使用。
